## 1. Что такое ThreatLens за одну фразу

**ThreatLens** — это веб-сервис, который проверяет на вирусы **два типа объектов**:
1. **Файлы** (exe, doc, zip, py…) — опасен или нет, и почему.
2. **Сетевой трафик** (файлы .pcap — это «запись» интернет-обмена компьютера) — есть ли там хакерская атака.

На обоих модулях **искусственный интеллект объясняет решение** человеческим языком (через YandexGPT).

**Живая демонстрация**: https://threatlens.tech

## 2. Архитектура — что-с-чем разговаривает

```
Пользователь
    ↓ (HTTPS)
  Nginx  ← веб-сервер, принимает запросы на 443 порту (https)
    ↓
  FastAPI (Python)   ← сам сервис, написан на Python, фреймворк FastAPI
    ↓                     ↓
Модуль файлов        Модуль сети (PCAP)
    ↓                     ↓
YARA + oletools       CICFlowMeter + ML-модели
    ↓                     ↓
SQLite-кэш            SHAP + YandexGPT
(SHA256 → результат)
```

**Что такое каждый кусок:**

- **Nginx** — «швейцар», принимает запрос из интернета, перенаправляет на Python.
- **FastAPI** — веб-фреймворк Python. Принимает POST-запросы с файлом, возвращает JSON с результатом.
- **SQLite** — маленькая встроенная база данных в одном файле. Хранит результаты сканов: если тот же файл уже сканировали — отдаёт из кэша мгновенно.
- **systemd** — служба Linux, которая держит наш Python-процесс запущенным (если упадёт — перезапустит).

**Где это всё крутится**: VPS (виртуальный сервер) на timeweb.cloud, Ubuntu Linux, IP `109.68.215.9`.

## 3. Модуль 1 — сканер файлов

### Что он делает
Принимает любой файл до 200 МБ, возвращает:
- **Risk score** от 0 до 100 (оценка риска).
- **Risk level**: LOW / MEDIUM / HIGH / CRITICAL.
- **Findings** — список конкретных подозрительных штук («содержит скрытый макрос», «обфусцированный код» и т.д.).
- **Recommendations** — что делать (удалить, не запускать, запустить в песочнице…).
- **AI explanation** (если включили галочку) — YandexGPT объясняет на русском, почему файл подозрительный.

### Как работает внутри
Для каждого файла **параллельно** запускается несколько анализаторов (это называется «multi-engine scan»):

1. **Generic-анализатор** — считает базовые штуки:
   - **MD5** и **SHA256** — цифровые «отпечатки пальцев» файла. Одинаковые байты → одинаковый хэш. Используется для сравнения с чёрными списками и для кэша.
   - **Энтропия** — мера «хаотичности» байтов. Формула: Шеннон, значения 0–8.
     - ~4 — обычный текст
     - ~6 — исполняемый файл
     - >7.5 — сжатый или зашифрованный файл (подозрительно, часто маскируют вирусы)
   - Определяет тип файла по «магическим байтам» (первые 4 байта), а не по расширению.

2. **YARA** — язык правил для поиска паттернов в файлах. Как регулярные выражения, но заточен под бинарные файлы. У нас около 300 community-правил (скачиваются при установке). Пример правила: «если внутри файла есть строка CreateRemoteThread И строка WriteProcessMemory — это скорее всего malware-инжектор».

3. **PE-анализатор** (для .exe, .dll) — [PE = Portable Executable, формат Windows-исполняемых файлов]. Смотрит:
   - Подписан ли файл (цифровой сертификат).
   - Какие Windows API-функции импортирует (если VirtualAlloc+WriteProcessMemory — код инжектит себя в другой процесс).
   - Много ли секций высокой энтропии (признак упаковщика — программа «сжата» чтобы обойти антивирусы).

4. **Office-анализатор** (для .doc, .xls, .docx) — использует библиотеку **oletools**. Ищет **VBA-макросы** и **auto-exec triggers** (макросы, которые запускаются автоматически при открытии документа). Это классическая техника фишинга: жертва открывает «накладную» → макрос тянет вирус из интернета.

5. **Script-анализатор** (для .py, .ps1, .bat, .js) — ищет опасные конструкции: `eval()`, `exec()`, `base64.decode(...)` (замаскированный код), скачивание файлов из интернета, запуск cmd/bash.

6. **Archive-анализатор** — умеет распаковывать ZIP/RAR/7z, включая **запароленные архивы** (если пароль задали в UI). Рекурсивно сканирует содержимое. Если архив запаролен + содержит .exe — это ОЧЕНЬ типичная схема malware, автоматически маркируется CRITICAL.

### Scoring — как собирается итоговая оценка
Все находки складываются по **весам** (threat_scorer.py). Например:
- Код-инжекция (CreateRemoteThread) = +40 баллов
- Отсутствие цифровой подписи = +5
- Высокая энтропия секций = +15
- YARA-совпадение с известным malware-семейством = +50
- Автозапуск макроса = +30
- Запароленный архив с .exe внутри = +50

Сумма → risk_score (0–100) → risk_level (LOW/MED/HIGH/CRITICAL).

### AI-объяснение
Если пользователь поставил галочку «AI-объяснение», все findings скармливаются **YandexGPT** (российский ChatGPT-аналог) с промптом вида:
> «Проанализируй эти индикаторы вредоносного ПО и объясни обычным языком, что файл делает и опасен ли он. Ответ на русском, без Markdown».

Отдаёт ~1000–1500 символов объяснения.

## 4. Модуль 2 — анализ сетевого трафика (PCAP)

### Что такое PCAP
**PCAP** (Packet Capture) — это **дамп сетевого трафика**. Программы типа Wireshark/tcpdump записывают КАЖДЫЙ пакет, который идёт по сети, в файл .pcap. Оттуда можно восстановить: кто с кем общался, когда, какие протоколы (TCP/UDP/ICMP), сколько данных передали.

### Что делает модуль
Пользователь грузит .pcap → система извлекает **потоки** → классифицирует каждый поток через ML → показывает таблицу + графики.

**Поток** (flow) = все пакеты между одной парой (IP-адрес:порт источника ↔ IP-адрес:порт назначения) за определённое время. Например, один HTTP-запрос = один поток.

### Пайплайн обработки
```
.pcap → CICFlowMeter → 80 статистических признаков на поток → ML-модель → класс + вероятность
                                                              ↘ SHAP → веса признаков
                                                              ↘ YandexGPT → текст
```

1. **CICFlowMeter** — это Python-библиотека, которая читает PCAP и для каждого потока считает **80 статистических признаков**:
   - Длительность потока
   - Количество пакетов в каждую сторону
   - Средний размер пакета, стандартное отклонение, минимум, максимум
   - **Inter-arrival time** (IAT) — сколько миллисекунд между пакетами
   - Количество TCP-флагов каждого типа (SYN, ACK, FIN, RST, PUSH)
   - Bytes/sec, packets/sec
   - …и ещё ~60 похожих метрик.

   Это не мы придумали — это **стандартный feature set** из датасета CIC-IDS2017 (см. ниже). На этих же 80 фичах обучают модели сотни исследователей по всему миру.

2. **ML-модели** — три штуки, обучены на датасете **CIC-IDS2017**.

### Что такое CIC-IDS2017 — САМОЕ ВАЖНОЕ для защиты
**CIC-IDS2017** — публичный датасет от Canadian Institute for Cybersecurity. Это **5 дней реального сетевого трафика** с лаборатории, где специально запускали **8 классов атак**:
- **DoS / DDoS** — заваливание сервера запросами (Slowloris, Hulk, GoldenEye, Slow HTTP Test, LOIC)
- **PortScan** — хакер сканирует порты жертвы, ища открытые
- **Brute Force** — перебор паролей (SSH, FTP)
- **Web Attack** — SQL-инъекции, XSS, brute force на логины
- **Infiltration** — заражение машины изнутри (dropper)
- **Botnet** — заражённые машины общаются с C&C (command-and-control)
- **Heartbleed** — эксплоит OpenSSL 2014 года
- **BENIGN** — обычный «нормальный» трафик (web-серфинг, почта, SSH…)

**Почему это важно**: это **эталонный датасет** в IDS-исследованиях (IDS = Intrusion Detection System, система обнаружения вторжений). Все топ-статьи (IEEE, ACM) используют именно его для сравнения. 2.8 млн сетевых потоков, всё размечено.

### Наши 3 модели
Обучаются скриптом `scripts/train_cicids.py` на этом датасете:

1. **Random Forest** (случайный лес) — классика ML. Это ансамбль из 100 **деревьев решений**. Каждое дерево — серия вопросов «если средний размер пакета > 500, иди налево, иначе направо». Голосование большинством. F1-score = 0.997.

2. **XGBoost** (eXtreme Gradient Boosting) — тоже ансамбль деревьев, но обучается по-другому: каждое следующее дерево **исправляет ошибки** предыдущего. Обычно чуть точнее RF. F1-score = 0.998.

3. **IsolationForest** — для **обнаружения аномалий без меток**. Не знает, какие атаки есть — просто говорит «этот поток сильно отличается от нормы». Полезно для **zero-day** атак, которых не было в обучении.

**F1-score = 0.998** значит: на 1000 потоков модель ошибается примерно 2 раза. Почти идеально.

### SHAP — объяснимый AI (Explainable AI, XAI)
**SHAP** = SHapley Additive exPlanations. Берёт из **теории игр** (формула Шепли из 1953 года, Нобелевка по экономике).

Зачем: ML-модели — «чёрный ящик». Сказала «это DDoS» — а почему? SHAP отвечает: «потому что packet rate был +3.2σ выше нормы (+0.45 к score), а средний размер пакета был маленький (+0.2), а длительность потока короткая (+0.15)…»

На UI это **waterfall-график**: красные полоски — признаки «за» атаку, синие — «против». Сумма = финальное решение.

Технически: `shap.TreeExplainer(model)` работает только с ансамблями деревьев (RF, XGBoost). Для каждого предсказания считает вклад каждой из 80 фич.

### Per-flow AI (YandexGPT)
Пользователь кликает «AI» на строке потока → промпт вида:
> «Классификатор определил поток как PortScan с confidence 98%. Фичи потока: packet_rate=5000/s, syn_count=450, avg_pkt_size=60. Объясни, что это значит, на русском».

YandexGPT отдаёт 500–1000 символов текста.

## 5. Веб-интерфейс — 5 страниц

1. **Сканировать** — загрузка файла, результат скана.
2. **Сеть (PCAP)** — загрузка pcap, таблица потоков, графики.
3. **История** — последние 50 сканов из SQLite.
4. **Поиск по хешу** — ввести SHA256, найти в кэше (без повторного скана).
5. **Статистика** — сколько файлов, сколько КРИТИЧЕСКИХ, сколько из кэша.

Всё — один HTML-файл (`index.html`) + один JS-файл (`app.js`) + FastAPI endpoints (`app.py`). Без React/Vue — чтобы проще.

**Графики** — библиотека **Plotly.js** (JavaScript, интерактивные — можно hover, zoom, export в PNG).

## 7. Что НЕ заслуга фреймворка, а наша работа

Комиссия может спросить «ну вы просто собрали open-source библиотеки». Ответ:

1. **Интеграция в единый pipeline** — YARA + oletools + PE + archive + YandexGPT работают параллельно через ThreadPoolExecutor, возвращают единый `AnalysisResult`. Это не 5 отдельных тулов, это ОДНА система.
2. **Обучение ML-моделей** — сами тренировали на 2.8 млн потоков CIC-IDS2017. Перебирали гиперпараметры, делали K-fold cross-validation (5 фолдов), сравнивали baseline (logistic regression) vs RF vs XGBoost. См. `scripts/train_cicids.py`, `scripts/compare_baselines.py`.
3. **Web-UI на SPA** — single-page-app на чистом JS, 5 разделов, реал-тайм графики. Без шаблонов-конструкторов.
4. **Production deployment** — не ноутбук в IDE, а живой сервис на HTTPS-домене, с systemd, nginx, SSL. Отказоустойчивый.
5. **Кэш SHA256** — если файл уже сканировали, результат возвращается за 5 мс вместо 2 секунд. Это реальная оптимизация.
6. **Объяснимость**: SHAP + GPT — два разных уровня объяснения одного предсказания. SHAP — для эксперта (какие фичи), GPT — для пользователя (естественный язык).

## 9. Q&A — типичные вопросы комиссии и как отвечать

### «Почему две модели (RF + XGBoost)? Достаточно одной»
Для **сравнения и валидации**. Если обе дают одинаковый ответ — уверенность выше. Плюс XGBoost чуть точнее, но RF **интерпретируемее** — можно вытащить feature importance глобально. Пользователь выбирает в UI.

### «Почему не глубокое обучение (нейронки)?»
На табличных данных (фичи — числа) деревья **стабильно бьют нейронки**, это подтверждено сотнями исследований (см. NeurIPS 2022 Grinsztajn et al.). Плюс: SHAP для деревьев работает быстро и точно, для нейронок — приблизительно. Плюс: модель 12 МБ, а не 2 ГБ.

### «Почему именно CIC-IDS2017, а не что-то новее?»
Это **самый цитируемый датасет** в IDS-литературе (>3000 цитирований на Google Scholar). Есть более новые (CIC-IDS2018, CSE-CIC-IDS2018), но 2017 — де-факто стандарт для сравнения алгоритмов. Фичи одинаковые, модель можно дообучать на новых данных.

### «Что если атака не из обучения? Zero-day?»
Для этого есть **IsolationForest** — аномалия без меток. Если Random Forest говорит «BENIGN с низкой confidence» + IsolationForest говорит «аномалия» — повод присмотреться.

### «YandexGPT же может галлюцинировать, как вы доверяете его ответам?»
AI-объяснение — это **дополнение**, не замена решения. Финальный вердикт даёт детерминированный классификатор (RF/XGBoost) + YARA. GPT только **переводит** findings на понятный язык. Если он напишет чушь — это видно по подсветке основных findings рядом.

### «Почему Python, а не Go/Rust/C++?»
Весь ML-стек (scikit-learn, xgboost, shap) нативно на Python. Экосистема. FastAPI (Python) + uvicorn держит 1000+ RPS. Для нашей задачи скорость Python не узкое место — узкое место это чтение файла + inference модели (GIL не мешает).

### «SQLite не масштабируется»
Для 1 узла и write-rate <100/s — вполне норм. Если надо масштабировать — заменить на Postgres (15 минут работы). **Архитектурно готово**: весь доступ к кэшу в одном файле `cache.py`.

### «Безопасность — вас самих не поломают?»
- Загруженные файлы сохраняются в `/tmp`, после скана удаляются.
- Максимум 200 МБ — нет риска забить диск.
- **Файлы НЕ запускаются** — только статический анализ (читаем байты, не исполняем).
- SSL (HTTPS), не HTTP.
- Rate limiting в FastAPI.
- Nginx перед Python — первая линия защиты.

### «А реально ли работает на проде?»
Да, открой `https://threatlens.tech` — боевой сервис. Не Heroku-демо, а VPS с systemd, nginx, SSL. Можно загрузить любой файл прямо сейчас.

### «Что было сложнее всего?»
- **Обучение моделей**: сырой CIC-IDS2017 имеет артефакты (дубликаты, незакрытые потоки, infinity в фичах). Надо было написать feature_pipeline (очистка NaN, удаление Inf, нормализация). 2 недели debugging.
- **Интеграция CICFlowMeter с scapy**: библиотека имеет баг (monkey-patch для работы с нашим scapy).
- **SHAP waterfall для 80 фич**: топ-10 по |значение| + остальные в «other», иначе график нечитаем.

### «Есть ли аналоги?»
- **VirusTotal** — SaaS, но закрытый, платный API, не объясняет решение.
- **Wireshark** — только визуализация пакетов, не ML.
- **Snort/Suricata** — сигнатурные IDS, не ML, не объясняют.
- **Наш проект** = open-source локальная альтернатива VirusTotal + ML-IDS в одной коробке + XAI.
