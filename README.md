---

ThreatLens combines classical static analysis (YARA, heuristics, PE/Office/script parsing) with a supervised ML pipeline for network flow classification. The current network model is trained end-to-end on Python-extracted flows (synthetic + CTU-13 botnet captures) — F1 0.96 / recall 96 % on real-world PCAPs the previous CIC-IDS2017-only model managed 0.86 % recall on. Every detection is accompanied by a human-readable explanation — either rule-based, generated by YandexGPT, or derived from SHAP feature attributions.

## File Analyzer

- **Static Analysis** &mdash; PE imports, strings, entropy, packer detection
- **Script Analysis** &mdash; Python, JavaScript, PowerShell, Batch, VBScript
- **Office Analysis** &mdash; VBA macros, OLE objects, DDE attacks
- **Archive Analysis** &mdash; ZIP, RAR, 7z, tar.gz &mdash; recursive scan, shows which file inside is dangerous
- **GitHub Repo Scan** &mdash; Scan any public repository for malicious code
- **1500+ YARA Rules** &mdash; Community rules + custom ThreatLens rules
- **Heuristic Engine** &mdash; Behavioral pattern matching for stealer / RAT / ransomware / miner / dropper / keylogger profiles
- **AI Explanations** &mdash; Built-in (no API needed) + optional YandexGPT
- **SHA256 Cache** &mdash; Instant results for previously scanned files

## Network Intrusion Detection

5-layer defense-in-depth pipeline. Trained end-to-end on Python-extracted flows so there is **zero feature drift between training and inference** &mdash; the single biggest reason the previous CIC-IDS2017-only model failed on real-world PCAPs.

### Pipeline

```
  PCAP -> Python FlowExtractor -> 81 features (70 CIC + 8 spectral + 3 YARA)
                                                         |
                                                         v
                                              XGBoost (7 classes)
                                                         |
                                                         v
                                         Mahalanobis abstainer (optional)
                                                         |
                                       +-----------------+----------------+
                                       v                                  v
                              SHAP explanation                   YandexGPT explanation
                              (per-flow top features)            (natural language, RU)
```

Full architecture writeup: [docs/architecture.md](docs/architecture.md).

### Real-world evaluation

The model is evaluated on **9 third-party PCAPs that were never seen during training** (Stratosphere SLIPS test captures + 30 % CTU-13 botnet hold-out + Wireshark sample captures for benign protocols). 693 flows total: 600 ATTACK + 93 BENIGN.

| Model | Recall ATTACK | Precision | F1 ATTACK | Specificity (BENIGN correctly NOT flagged) |
|---|---:|---:|---:|---:|
| Previous prod (`results/cicids2017/`, Java-extracted train) | 0.86 % | 60.0 % | 0.017 | 97.8 % |
| **Current prod (`results/python_only/`, Day 9e)** | **96.25 %** | **95.98 %** | **0.961** | **84.9 %** |
| Improvement | **&times;112** | **+35.9 pt** | **&times;57** | (specificity dropped because the model is now actually trying to flag attacks) |

CTU-13 botnet hold-out (253 flows, all ATTACK by construction): **100 % recall** vs 5.93 % on the previous model.

| Capture | Flows | True ATK | Detected | Recall |
|---|---:|---:|---:|---:|
| `slips_test7_malicious.pcap` (Stratosphere mixed C2) | 279 | 279 | 273 | 97.8 % |
| `slips_ssh-bruteforce.pcap` (TCP/902 brute-force) | 67 | 67 | 60 | 89.6 % |
| `slips_test8_malicious.pcap` | 1 | 1 | 1 | 100 % |
| `wireshark_tls12-chacha20.pcap` (modern TLS) | 7 | 0 | 0 ✓ | n/a |
| `wireshark_http2-*.pcap` (HTTP/2 multiplexed) | 2 | 0 | 0 ✓ | n/a |
| `wireshark_dns-mdns.pcap` (mDNS / DHCP / NTP / multicast LAN noise) | 83 | 0 | 14 (FP) | n/a |
| `wireshark_tls-renegotiation.pcap` | 1 | 0 | 0 ✓ | n/a |

Reproduce: `python scripts/eval_python_only.py`.

### Time-validation: fresh 2024-2025 malware

The 96.25 % above is on PCAPs from &le;2018 captures. To check the model is not
quietly stale, we live-ingest modern malware on the day evaluation runs
(Stratosphere CTU 2024 + malware-traffic-analysis.net 2025): 25 PCAPs,
**5011 attack flows**, families covered: Lumma, StealC, Formbook/XLoader,
Rhadamanthys, NetSupport RAT, MacSync, ClickFix, Kongtuke.

| Test set | Year | N PCAPs | N attack flows | Recall (any-attack) | Abstain rate |
|---|---:|---:|---:|---:|---:|
| Historical real-world (above) | &le;2018 | 7 | 600 | **96.25 %** | n/a |
| Fresh sandbox ingest | 2024-2025 | 25 | 5011 | **72.24 %** | 70.8 % |

Honest read of the drop:

- **Per-source:** Stratosphere CTU 2024 &rarr; 74.8 %, MTA 2025 &rarr; 47.1 %. Model is weaker on the freshest channel &mdash; the reverse of what a benchmark-fitted model would show.
- **Per-family:** strong on infrastructure-shaped C2 (NetSupport 100 %, StealC 82 %, ClickFix 78 %, Kongtuke 75 %), weak on short-lived stealer infection traffic (Lumma 40 %, Formbook 27 %, Rhadamanthys 25 %). Several PCAPs only contain 1&ndash;3 flows of "initial check-in" that look like normal HTTPS.
- **Abstainer flagged 70.8 %** of these flows as out-of-distribution &mdash; selective prediction is doing its job, the operator sees "uncertain" on most of this slice rather than blind classification.

Sandbox PCAPs are 100 % labeled `Bot`, so we report recall only &mdash; there is no benign traffic in this set, so precision against a clean baseline is not computable. Per-capture / per-family breakdown: [`results/python_only/sandbox_eval.json`](results/python_only/sandbox_eval.json). Reproduce:

```bash
python scripts/ingest_sandbox_pcaps.py --source all --limit 100
python scripts/extract_sandbox_pcaps.py --skip-pcaps "CTU-Malware-Capture-Botnet-61-1__bot__Bot.pcap"
python scripts/eval_sandbox.py
```

### Operating-mode trade-off (workload reduction)

The Mahalanobis abstainer (`results/python_only/mahalanobis_abstainer.joblib`) lets operators trade attack recall for false-positive rate without retraining. Measured on the same 693-flow real-world test set:

| Mode | Auto-classified | Review queue | FP rate | Recall (model alone) | Recall (with reviewer) |
|---|---:|---:|---:|---:|---:|
| **lenient** (default) | **100.0 %** | 0.0 % | 2.02 % | **97.83 %** | 97.83 % |
| safe (cov 0.95) | 81.8 % | 18.2 % | 2.47 % | 80.17 % | 99.33 % |
| strict (cov 0.80) | 67.5 % | 32.5 % | 2.35 % | 64.50 % | 99.67 % |
| paranoid (cov 0.50) | 30.4 % | 69.6 % | 0.47 % | 23.50 % | 99.83 % |

Toggle modes per deployment via `THREATLENS_STRICT_MODE=1`. Reproduce: `python scripts/workload_metric.py`.

### Attack classes (current model)

`BENIGN` | `Bot` | `DoS Hulk` | `DoS slowloris` | `FTP-Patator` | `PortScan` | `SSH-Patator`

The legacy `cicids2017` model recognised 14 classes (including `Heartbleed`, `DDoS`, `Web Attack` variants and `Infiltration`). Those classes are still selectable via `THREATLENS_ML_DIR=results/cicids2017` for backwards compatibility but are **not recommended for real-world traffic** &mdash; their training data was Java-extracted and exhibited the cross-extractor drift that drove zero-shot recall to 0.86 %.

### How we got here

The 0.86 % &rarr; 96.25 % recall jump came from a 9-day improvement plan (see [docs/improvement_plan.md](docs/improvement_plan.md)):

| Day | Change | Real-world recall |
|---:|---|---:|
| 0 | Baseline: CIC-IDS2017 train, Java-extracted features | 0.86 % |
| 4 | + YARA-on-payload features | 0.86 % |
| 5 | + CIC-IDS2018 retrain (cross-dataset gap closed) | 0.86 % (still drifted) |
| 7 | Diagnostic: 91-98 % of features have p<0.001 between Java/Python extractors | &mdash; |
| 8 | Mahalanobis abstainer (Day 8 selective prediction) | 11 % at 50 % coverage |
| 9 | **Drop Java CSVs entirely &mdash; train only on Python-extracted flows** | **92.5 %** |
| 9b/d | Diverse benign (mDNS / DHCP / NTP) + port-diverse SSH/FTP brute-force | 93.1 % |
| 9e | Long-tail: TLS quick-fetch + HTTP/2 + TCP-micro brute-force | **96.25 %** |

Per-day writeups: [day4](docs/day4_yara_synergy.md), [day5](docs/day5_combined_retrain.md), [day6](docs/day6_real_world_reeval.md), [day7](docs/day7_drift_diagnostic.md), [day8](docs/day8_selective_prediction.md), [day9](docs/day9_python_only_retrain.md), [day10](docs/day10_web_app_integration.md).

### Reproducing results

```bash
# Install deps
pip install -r requirements.txt

# Generate the Python-extracted training data (~3 min total)
python scripts/generate_diverse_benign.py --flows-per-profile 400
python scripts/generate_attack_volume.py --attempts-per-profile 400
python scripts/generate_long_tail.py --flows-per-profile 500
python scripts/extract_diverse_benign.py
python scripts/extract_attack_volume.py
python scripts/extract_long_tail.py

# Train (10 sec) + fit abstainer (2 sec)
python scripts/train_python_only.py
python scripts/fit_selective_python_only.py --target-coverage 0.99

# Evaluate on real-world PCAPs (1 sec, uses cached flow extraction)
python scripts/eval_python_only.py
python scripts/workload_metric.py
```

All scripts use `random_state=42`. The training data PCAPs and resulting model artefacts are version-controlled (see `.gitignore` whitelist for `results/python_only/`).

The legacy CIC-IDS2017 reproducible recipe (still works for the 14-class model) is preserved in [`threatlens/network/README.md`](threatlens/network/README.md).

### Docker

For reviewers who want to skip the local Python setup:

```bash
docker build -t threatlens .
docker run --rm -p 8888:8888 threatlens
# → http://localhost:8888    (UI)
# → http://localhost:8888/health    (= {"status":"ok","7z_available":true})
```

Or via compose (mounts `./scans` for CLI usage and reads `YANDEX_OAUTH_TOKEN` from `.env`):

```bash
docker-compose up --build
```

Build context is ~30 MB (`data/` and the legacy `combined_v2/` model bundle are excluded by `.dockerignore`); resulting image is ~2.5 GB (mostly numpy / scipy / xgboost / scikit / matplotlib + 566 community YARA rules). The image ships the Day 9e `python_only` model as default — override with `-e THREATLENS_ML_DIR=/app/results/cicids2017` to roll back to the legacy bundle.

Verified inside the built image: `GET /` returns 200, `GET /health` returns `{"status":"ok","7z_available":true}`, and `joblib.load("/app/results/python_only/xgboost.joblib")` produces the same 7-class / 65-feature classifier used in the published metrics.

**Why Python 3.12 in Docker but 3.14 in dev:** `yara-python` 4.5.x has no Python 3.14 wheel as of 2026-04. Docker pins 3.12 so YARA-on-payload features actually fire — they get dropped by the variance filter in our local training because the dev box can't `import yara` (see `docs/day4_yara_synergy.md`). Anyone re-training inside the container can re-enable that feature column.

### Stats API

`GET /api/stats` exposes aggregate usage telemetry suitable for a public
dashboard. Response is cached in memory for 60 seconds. Individual scan
records are never exposed — only totals, 7d / 24h windows and a 30-day
daily histogram.

```json
{
  "total_scans": 0,
  "scans_last_7d": 0,
  "scans_last_24h": 0,
  "unique_users_total": 0,
  "unique_users_7d": 0,
  "threats_detected_total": 0,
  "scans_by_type": {"file": 0, "pcap": 0, "hash_lookup": 0},
  "avg_scan_duration_ms": 0.0,
  "first_scan_at": null,
  "daily_scans_last_30d": []
}
```

> The numbers above are placeholders showing the response schema. Real values
> are returned live by `/api/stats` on the running instance.

Unique users are counted via `sha256(STATS_SALT + client_ip)` — the raw
IP is never persisted.

**`STATS_SALT` must be set in the environment** (systemd drop-in / `.env`).
If it is missing, the app logs a `WARNING` and falls back to a
per-process random salt via `secrets.token_hex(16)`; this keeps hashes
unpredictable but resets the unique-user counter on every restart.

`GET /api/cache-stats` (previously `/api/stats`) keeps the legacy cache
metrics (unique files, cache hits, risk breakdown) used by the internal
"Статистика" page.
