# Real-world evaluation — honest numbers

**Run date:** 2026-04-22
**Model under test:** `xgboost.joblib` (primary), trained on 50 000 stratified
CIC-IDS2017 flows with `random_state=42`. See
[`results/cicids2017/metrics.json`](../results/cicids2017/metrics.json).
**Reported F1 on CIC-IDS2017 test split:** **0.9983** (5-fold CV 0.9983 ± 0.0003).
**This document measures the F1 on a different distribution.**

## Why this evaluation exists

The README already acknowledges feature drift between the Java CICFlowMeter
that generated CIC-IDS2017 and the Python flow extractor used at inference
time. This document replaces that acknowledgement with **numbers**, using
PCAPs from two sources independent of CIC-IDS2017:

- [Stratosphere IPS / Slips](https://github.com/stratosphereips/StratosphereLinuxIPS/tree/master/dataset) — 4 labeled attack captures
- [Wireshark test captures](https://github.com/wireshark/wireshark/tree/master/test/captures) — 5 benign protocol samples

See [`data/real_pcap/README.md`](../data/real_pcap/README.md) for file-level
provenance. Reproduce with:

```bash
python scripts/eval_real_world.py
# writes results/real_world_eval.json
```

## Headline result

| Metric | CIC-IDS2017 test split | Real-world eval (this doc) |
|--------|----------------------:|---------------------------:|
| Weighted F1 (all classes) | **0.9983** | **0.4273** (strict subset) |
| Binary F1 (ATTACK class) | — | **0.0000** |
| Binary accuracy | — | **0.2114** |

**The production model misses every attack in the real-world set.** Of 347
flows extracted from known-attack PCAPs, zero were labeled as any attack
class — all were classified `BENIGN`.

This matches the drift hypothesis in the README: the 99.83% on the CIC-IDS2017
test split does not transfer to third-party captures.

## What this means

1. **Specificity on benign traffic is preserved.** All 93 flows from Wireshark
   benign captures classify as BENIGN — no false positives on out-of-distribution
   benign samples. This is a real strength.
2. **Sensitivity on attacks is zero.** The model does not recognise SSH
   brute-force, UDP-based C2, or mixed malicious traffic from Slips. Detection
   performance collapses when the attack distribution differs from CIC-IDS2017.
3. **The ICMP port-scan produced 0 flows** because the flow extractor
   (`threatlens/network/flow_extractor.py`) aggregates TCP/UDP packets only.
   ICMP-based scans are silently dropped before they reach the classifier.
   This is a second, orthogonal gap — not a model error but an extraction gap.

## Hypotheses for the 0.998 → 0.00 drop

These are hypotheses, not conclusions — to be confirmed in follow-up work:

1. **Feature computation drift.** The Python extractor does not produce
   bit-identical features to Java CICFlowMeter. In particular, flag-count
   columns (SYN, PSH, URG, ECE, CWE) and inter-arrival time statistics
   differ in edge cases. A model that overfits to these subtle cues would
   degrade this sharply.
2. **Shorter real-world flows.** CIC-IDS2017 attack flows often last seconds
   to minutes (Patator, Slowloris). Slips SSH brute-force produces shorter,
   more bursty flows (67 flows across 1178 packets). Features like `Idle Mean`
   and `Flow Duration` (the top-5 predictive features) operate in a range the
   model never saw.
3. **Different attacker tooling.** CIC-IDS2017 SSH-Patator was generated by
   the `Patator` script against a specific victim topology. Slips used its
   own lab setup. Even at the same "class" label, the statistical
   fingerprint differs.
4. **Scale mismatch.** Several Wireshark benign PCAPs produced 1–7 flows,
   each extremely short. These sit near the `BENIGN` centroid by default
   and would classify as BENIGN regardless of content — so the 100% benign
   match rate is partly a low-information artefact, not model merit.

## What would move the needle

Ordered by expected impact vs effort:

1. **Re-extract CIC-IDS2017 PCAPs with our own Python extractor** and retrain.
   This collapses the training/inference mismatch into a single toolchain.
   If F1 stays high after this, drift was the cause; if it drops, the feature
   set itself is brittle.
2. **Add CIC-IDS2018 or UNSW-NB15 as a second training source.** Cross-dataset
   training explicitly optimises for generalisation rather than dataset-specific
   cues.
3. **Include flow duration normalisation or log-transform** on the top-5
   features so the model is less sensitive to absolute flow length.
4. **Add ICMP support to the flow extractor** or document explicitly that
   ICMP-based scans are out of scope.
5. **Calibrate per-class thresholds.** Current predictions are argmax of
   the softmax. If SSH-Patator scores even 0.1 probability on a real
   brute-force flow, a lower threshold may surface it — at the cost of
   higher false-positive rate, which is currently 0. This is the
   cheapest lever before retraining.

## Honesty disclaimer

This evaluation has real limits:

- **Only 440 flows** (347 attack + 93 benign). Any single number here has
  wide confidence intervals; we report point estimates without error bars.
- **Capture-level labels only.** We treat every flow inside an "attack"
  PCAP as attack. Some of those flows are almost certainly background
  BENIGN traffic. That inflates the "false negative" count — the real
  miss rate on actual attack flows might be lower. But given the model
  detected **zero** attack flows, the direction of the finding is robust:
  detection sensitivity on out-of-distribution attacks is poor.
- **No adversarial evaluation.** We did not test evasion-crafted traffic,
  only off-the-shelf labeled datasets.
